# ia-analyses-db 架構指南

## 速覽
- 定位：IA Analyses 的 PostgreSQL runtime、schema contract、patch、seed、backup / restore 主權 repo
- 主要入口：`make dev-env` → `make dev-up` → `make dev-migrate` → `make dev-smoke-analytics`
- 依賴：被 `ia-analyses-go` 依賴（提供 DB runtime 與 schema）
- 類型：容器型 | DB

## Repo 定位

這個 repo 是 IA Analyses 的 DB 主權 repo，負責 PostgreSQL runtime、schema 初始化、patch 套用、seed、backup / restore，以及 baseline 驗證。Go pipeline 已轉由 `ia-analyses-go` 作為主要操作入口；本 repo 只保留 bridge copy 以利過渡期對照。

## 架構分層

### 1. 環境與容器層

- `Makefile`：人類操作入口，正式命名是 `dev-*` / `prod-*`
- `.env.dev`、`.env.prod`：環境來源檔
- `.env`：執行當下使用的環境檔，由 `make dev-env` 或 `make prod-env` 派生
- `docker-compose.yml`：啟動 PostgreSQL 與 Adminer

### 2. DB 結構層

- `db/init/001_schema.sql`：初始化 schema、seed 與核心 constraint
- `db/patches/`：正式 patch SQL
- `db/migrations_draft/`、`db/validation_draft/`：草稿中的結構演進與驗證設計

### 3. DB 維運層

- `scripts/db_backup.sh`、`scripts/db_restore.sh`：一般 backup / restore
- `scripts/db_restore_baseline.sh`：baseline restore
- `scripts/db_apply_patches.sh`：套用 patch
- `scripts/db_smoke_analytics.sh`：最小 analytics 驗證
- `scripts/db_size.sh`：查看資料庫大小

### 4. 開發測試用（bridge copy）

- `cmd/`、`internal/`：保留原始 Go slice 與 package 對照
- Makefile 的「開發測試用」群組保留 `sales-pipe-*` 與 `sync-sales-dims*`，僅供 bridge copy 對照與開發測試使用
- 這些 target 不是正式操作入口；正式操作請使用 `ia-analyses-go`
- 舊有的 `sync-athena-*` 殭屍 target 已於本輪清除

## 核心資料模型

目前核心資料模型由 1 張 owner 對照表、3 張維度表、1 張狀態語意表、1 張付款維度表、1 張銷售 fact 表組成：

- `ia_users`：外部 owner key 與內部 owner id 對照
- `pos_order_type_dim`：canonical 訂單型態
- `pos_payment_type_dim`：canonical 付款型態
- `pos_order_status_dim`：raw status 的語意歸類
- `pos_product_dim`：商品維度
- `pos_branch_dim`：門市維度
- `pos_sales_hourly_fact`：商品層級的小時銷售彙總事實表

`pos_sales_hourly_fact` 的責任是商品層級銷售分析，不承接 raw payment 對帳、完整訂單生命週期、condiment 明細與 branch opening。那些資料域若要進來，應由後續獨立 fact 承接。

## 執行流程

### 啟動 DB

1. `make dev-env`
2. `make dev-up`
3. `make dev-migrate`

### 還原與驗證

1. `make dev-restore BACKUP_FILE=...` 或 `make dev-restore-baseline BASELINE_FILE=...`
2. `make dev-smoke-analytics`
3. `make dev-backup`

### 與 Go pipeline 的互動

1. 本 repo 提供 PostgreSQL schema 與 runtime
2. `ia-analyses-go` 的 `sync-sales-dims` 先同步商品與門市維度
3. `ia-analyses-go` 的 `sales-pipe` 再把 hourly sales fact 寫入這裡
4. `ia-analyses-go` 的 `check-data` 會對 Athena source 與本 repo 內已落地資料做 read-only compare

## 維護原則

- 正式入口只以 `make dev-*` / `make prod-*` 為準
- `backup/` 內的 dump 是本機 restore 載體，不應提交實體 dump 到 git
- schema 事實來源以 `db/init/001_schema.sql` 與正式 patch 為準
- 若本 repo 的 schema 發生變更，需同步檢查 `ia-analyses-go` 內相依 package 與 workspace 關聯文件
