# ia-analyses-db 架構指南

最後檢視：2026-05-20

註：2026-05-19 起，正式 Makefile 入口改為 `dev-*` / `prod-*`。本文中的 `db-*` 若出現在歷史 phase 記錄，僅代表當時名稱，不再是正式操作介面。

## 目標

這個 repo 的角色不是 Athena raw mirror，而是分析層資料庫第一階段先解決三件事：

1. 資料庫容器與初始化流程
2. 分析用 schema 與固定 mapping table
3. `sync-athena` 的 CLI 邊界，先能做 dry-run 與參數驗證

## 邊界

第一階段明確不做以下事情：

- 不直接寫 Athena SQL
- 不直接連 PostgreSQL 寫入 fact/dim
- 不處理 owner user 自動建檔
- 不實作重跑、補資料、游標管理、lock table

這樣做的目的，是先把 repo 的 runtime 邊界穩住，再往內填同步細節

## Schema 設計摘要

### `ia_users`

- 功能：將外部 `owner_user_key` 正規化成內部整數鍵
- 原因：fact 表避免重複長字串；未來也能擴充 owner 級 metadata

### `pos_sales_hourly_fact`

- grain：`owner_user_id + business_date + hour_of_day + branch_id + product_no + order_type_id + payment_type_id`
- 指標：`qty_milli`, `gross_sales_milli`, `discount_milli`, `surcharge_milli`, `net_sales_milli`, `sales_ex_tax_milli`, `included_tax_milli`, `excluded_tax_milli`, `tax_milli`
- 口徑：`net_sales_milli` 代表含稅、折扣與加價後的實際認列銷售額；`sales_ex_tax_milli` 才是未稅銷售額
- 策略：預留日級 replace，而不是只做 upsert，避免來源 group 消失時殘留舊資料

### `pos_product_dim` / `pos_branch_dim`

- 用途：保留對人可讀的名稱
- 策略：先做 current snapshot 型維度，不急著引入 SCD

### `pos_order_type_dim` / `pos_payment_type_dim`

- 用途：把來源中不穩定的文字型態，收斂到固定 id
- 特點：這兩張表先由 seed 管理，後續實作 sync 時只做 lookup

- `pos_sales_hourly_fact.business_date` 在 Phase 2C-1 不改欄位名，但正式語意定稿為 `sale_period`。

### 為什麼不做單一萬用 fact

根據 QuickSight analysis inventory、metric mapping、fact gap analysis，目前已確認以下需求不應被混在同一張 fact：

- `sale_period` / `tr_date` / `t_open_date` 的雙日期以上語意
- 商品層銷售 vs 訂單層 order count / void lifecycle
- canonical payment type vs raw payment `name` / `memo1`
- condiment 明細
- branch opening / terminal 狀態

因此架構上改採多 fact 模型，而不是把所有報表需求堆回 `pos_sales_hourly_fact`。

### `pos_sales_hourly_fact` 的固定責任

- 只承接商品層級銷售彙總
- source path 只吃 status-aware `status = 1` latest row
- 支援產品銷售、杯數、稅、折扣、附加費、canonical order/payment 分析
- 不承接 raw payment 對帳、完整 order count、void lifecycle、condiment、branch opening

### 建議新增的 fact 邊界

- `pos_order_daily_fact`
	- 承接安全可聚合 `order_num`
	- 承接 `status = 1 / -2 / -1 / 2` 的訂單狀態統計
	- 承接 `sale_period`、`tr_date`、`t_open_date`、`void_sale_period`
	- 承接門店對帳單所需訂單層拆項
- `pos_payment_daily_fact`
	- 承接 raw payment `name` / `memo1`
	- 承接 `amount`、`change`、`amount - change`
	- 承接付款別報表與對帳付款側需求
- `pos_condiment_hourly_fact`
	- 承接 condiment group / name、qty、subtotal
	- 只服務調味報表與未來 modifier 分析
- `pos_branch_opening_daily_fact`
	- 承接 `sale_period`、`terminal_no`、`open_time`、`close_time`
	- 只服務門市營業時間類報表

### Date semantics 風險

- 現有 `business_date` 仍是模糊欄位名，但 QuickSight 實際上同時需要 `sale_period`、`tr_date`、`t_open_date`。
- 若在 sales fact 同時承接這三種日期，會讓 grain 與報表責任一起膨脹。
- Phase 2C-1 定稿：`pos_sales_hourly_fact.business_date` 只收斂為 `sale_period`；`tr_date` / `t_open_date` 交給 order/payment fact。

### QuickSight compatibility 原則

- 優先保留報表真正使用的 raw 維度，不要用 canonical dim 強行取代。
- `payment_type_id` 不等於 QuickSight payment `name` / `memo1`。
- `order_type_id` 不等於 QuickSight raw `destination`。
- `cate_no` / `cate_name` 已定稿補在 `pos_product_dim`，不應直接塞進 sales fact。
- condiment 與 branch opening 應維持獨立 fact，不應回寫到 sales fact。

### `internal/athena`

- 負責 Athena 查詢計畫、來源表清單、未來的 SQL builder 邊界
- 目前 Phase 2A dry-run 會真的呼叫 Athena，但仍只做 read-only preview，不寫 PostgreSQL
- Phase 2B 已把 dry-run 切成 fast / full 兩種模式
- fast mode 只跑 source metrics、result metrics、reconciliation summary、top tax delta sample、status excluded summary，適合 7 天、31 天長日期窗驗證
- full mode 另外保留 mapping、preview sample 與完整 debug 區塊，適合 1 到 3 天短窗 root cause investigation
- `order_additions` 目前已納入 discount/surcharge 分攤，`orders.included_tax_subtotal` 仍是 v1 的 tax truth
- Phase 2B 另外補了 reconciliation debug：把 source / preview 的 completed scope、no-items、zero-denominator、status 排除、`order_additions.include_tax` 比較、rounding delta 與 Top-N order sample 都顯式輸出
- 這些 debug 都只在 dry-run 顯示，不寫 PostgreSQL，也不改變正式 preview/fact 欄位型別；正式輸出仍維持 milli bigint

### Phase 2B status-aware source 邊界

- sales path：只吃 `status = 1` latest row
- void path：只保留 `status = -2` latest row
- excluded path：保留 `status = -1 / 2` 做 debug，不進 sales / void 主口徑
- 不使用全 status 混合的 latest row，避免同一個 `order_id + t_open_date` 的生命週期版本在 sales path 互相放大

### 長日期窗驗證建議

- 7 天與 31 天驗證請優先使用 fast mode，避免 full debug 的 trace 類查詢把長窗成本放大
- 目前 fast mode 驗證結果：
	- 7 天：result_aggregation 775,736，tax delta -9,297，net delta 365，sales_ex_tax delta 9,662，top tax delta 最大絕對值 6
	- 31 天：result_aggregation 3,698,110，tax delta -45,109，net delta 1,511，sales_ex_tax delta 46,620，top tax delta 最大絕對值 7
- 這兩個日期窗的最大 tax delta 都仍在單筆個位數 milli，表示主問題已收斂，不需要再用長窗 full debug 重複追查

### `internal/postgres`

- 負責 DSN、連線、交易與日後的 replace/upsert 寫入策略
- 本輪已開啟 `SQLSalesFactWriter` 的 local-only 實際用途，但仍只允許單日 `public.pos_sales_hourly_fact` day-level replace
- `DisabledSalesFactWriter` 仍是 validate-only / write-plan 的安全預設；只有 `--write-pg` + `--local-only-actual-write` 才會切到 `SQLSalesFactWriter`
- post-insert persisted target metrics 會在同一個 transaction 內讀取，確保 compare 發生在 commit 前而不是 commit 後
- Phase 2C-5.4 已驗證 rerun 仍會走 transaction 內 delete + insert，不會因為已有資料而累加，也不會跳過刷新 `updated_at`
